## Preparing WANDS for LLM relevance judging

Before benchmarking any models, I needed to turn WANDS into the kind of input my judging pipeline could handle.

WANDS is organized around three pieces of information: search queries, products, and human relevance labels. I joined these into a single query-product-label table where each row represented one product candidate for one search query.

For the LLM input, I kept the fields that looked directly useful for relevance judging:

- the search query
- product name
- product class
- category hierarchy
- product description

I removed fields like product ratings and review counts. Those fields may be useful for ranking products, but they are not direct evidence that a product matches a query.

| query_id | product_id | label | query | product_name | product_class | category hierarchy | product_description | score |
|---|---|---|---|---|---|---|---|---|
| 15 | 39828 | Irrelevant | black 5 drawer dresser by guilford | fairlin 5 drawer storage cabinet | Office Storage Cabinets | Furniture / Office Furniture / Office Storage Cabinets | the features never stop unfolding with this multifunctional storage hutch... | 0 |

*Table 1. A single benchmark row after joining the WANDS query, product, and label tables. Each row becomes a query-product relevance judgment.*

The original WANDS label strings were converted into numeric scores:

| WANDS label | Score |
|---|---:|
| Irrelevant | 0 |
| Partial | 1 |
| Exact | 2 |

This matched the output format expected by my judging pipeline. During training and benchmarking, models were asked to return only one of:

```text
SCORE: 0
SCORE: 1
SCORE: 2
```

The exported instruction-tuning rows contained an instruction, an input block, and a target output. The input block included the query and product details, while the output contained only the score.

| query_id | product_id | instruction | input | output |
|---|---|---|---|---|
| 15 | 39828 | Given a search query and a candidate product, predict the relevance score. Apply the scale strictly: 0 = Irrelevant — the product type or category does not match what the shopper searched for. 1 = Partial — correct general category but misses a key attribute from the query. 2 = Exact — the product type and all key attributes stated in the query are directly satisfied. Return exactly one line: SCORE: <number> | Query: black 5 drawer dresser by guilford Product name: fairlin 5 drawer storage cabinet Product class: Office Storage Cabinets Category hierarchy: Furniture / Office Furniture / Office Storage Cabinets Product description: the features never stop unfolding with this multifunctional storage hutch . three cubbies are created in the upper open portion of the cabinet along with three small wood drawers so you can get on top of your organization game . this modern storage cabinet can also function as a computer hub . the central drawer has a drop-down front and cord management in the back so you can create a home for your laptop where you can easily pull it out to use or tuck it away to charge . at the bottom of this library hutch , there are two doors with adjustable soft-close hinges for some closed storage options . wall anchoring hardware is also included . | SCORE: 0 |

*Table 2. Example instruction-tuning row. The model receives a query and product fields, then learns to return a score-only judgment.*

## Building the benchmark set

I did not use a purely random benchmark split. Random sampling would have made the benchmark easier to create, but less useful for diagnosis.

Instead, I built a controlled benchmark set:

- 50 held-out queries
- 10 products per query
- 500 total query-product pairs
- each benchmark query included at least two `Exact`, two `Partial`, and two `Irrelevant` examples

This made the benchmark more balanced than the natural WANDS distribution. The goal was to create a benchmark where every model had to distinguish between all three relevance levels.

![Train and benchmark label distribution](/assets/img/2026-05-02-judgement-ai/label_distribution_train_vs_benchmark.png)

*Figure 3. Label distribution in the training and benchmark sets. The benchmark was more balanced so that model behavior on all three labels could be inspected.*

The benchmark was also structured by query. Each query contributed 10 products, which made it easier to compare model behavior across different search intents.

![Benchmark label composition by query](/assets/img/2026-05-02-judgement-ai/benchmark_composition_by_query.png)

*Figure 4. Benchmark composition by query. Each held-out query has a controlled mix of Irrelevant, Partial, and Exact products.*

## Building the training set

For training, I sampled from the remaining queries after removing the benchmark queries. The training set was compact by design: large enough to teach the model the format and the relevance rubric, but small enough to make local fine-tuning practical.

This produced 5,074 training rows from 429 training queries, with a cap of 12 products per query.

The training data was capped per query so that a few large queries would not dominate the dataset. I also tried to rebalance the labels so that `Exact` and `Irrelevant` examples were better represented than they were in the raw dataset.

Although I tried to rebalance the labels, the final training set was still `Partial`-heavy: about 48.9% `Partial`, 28.1% `Irrelevant`, and 23.0% `Exact`. That became important later: the fine-tuned model often behaved conservatively, preferring `Partial` over `Exact`.

## Data validation

Before fine-tuning, I also used a small split script to validate the instruction-tuning JSONL. It checked that every row had the required fields, that outputs were restricted to `SCORE: 0`, `SCORE: 1`, or `SCORE: 2`, deduplicated rows, and split train/eval data by `query_id` rather than arbitrary rows.

This mattered because query-level splitting reduces leakage. If the same query appears in both training and evaluation, the benchmark can look better than it really is.

## Why this preparation mattered

The data preparation ended up shaping the entire experiment.

The benchmark was a controlled test set for relevance judging. That made it more useful for comparing LLM judges, but it also means the results should be interpreted carefully: this benchmark measures performance on a balanced, diagnostic WANDS subset, not on the raw WANDS distribution.

I wanted to know whether a model could reliably tell the difference between Irrelevant, Partial, and Exact products when all three were present.

## Fine-tuning a local judge

After preparing the training and benchmark datasets, I wanted to test whether a small local model could learn the WANDS relevance rubric.

The task was narrow: given a search query and a candidate product, return one of three relevance scores.

```text
SCORE: 0
SCORE: 1
SCORE: 2
```

I used `unsloth/qwen3.5-4b-base` as the base model. This was important because I was not starting from a polished instruction-following judge. The model had to learn both the relevance task and the strict output format from the fine-tuning data.

I trained the model in Google Colab using Unsloth Studio. Since I was working within the limits of a free Colab session, I used QLoRA rather than a full fine-tune. The goal was to see whether a practical local fine-tuning workflow could produce a useful search relevance judge.

The training setup was close to the defaults:

![Unsloth Studio training settings](/assets/img/2026-05-02-judgement-ai/unsloth_training_settings.png)

*Figure X. QLoRA training settings in Unsloth Studio. I mostly used the default configuration and limited training to one epoch.*

The run completed after 620 steps, which corresponded to one epoch over the training set. I watched the training loss, evaluation loss, learning rate, and gradient norm during the run. The evaluation loss decreased early and then mostly flattened, so I did not treat the loss curve as proof that the model had become a reliable judge. The real test would be the held-out benchmark.

![Unsloth Studio training progress](/assets/img/2026-05-02-judgement-ai/unsloth_training_progress.png)

*Figure X. Training progress near the end of the run. The experiment was practical: one epoch, no hyperparameter sweep, and a free Colab environment.*

After training, I exported a merged `bf16` model through the Hugging Face API. I then ran the fine-tuned model locally through Ollama and used `judgement-ai` to generate relevance judgments on the 500-row benchmark.

This part of the project made the limits of fine-tuning obvious. There were many choices that could have changed the outcome: the base model, the train/benchmark split, the label distribution, context length, learning rate, number of epochs, LoRA rank, and even the exact prompt format.

I did not try to optimize all of those. The goal was to complete a realistic workflow:

`WANDS dataset → prepared JSONL → QLoRA fine-tune → local Ollama model → judgement-ai benchmark → metric comparison`

That made the result useful even though the fine-tuned model was not a human-level judge.

## What the results were

After fine-tuning, I used `judgement-ai` to run the same 500-row benchmark across several judges:

- instruct bf16 Qwen 3.5 4B
- my fine-tuned bf16 Qwen 3.5 4B
- GPT-5.5

Each model produced one score per query-product pair, and I compared those scores against the original WANDS human labels.

### Overall model performance

![Model agreement metrics](/assets/img/2026-05-02-judgement-ai/model_agreement_metrics.png)

*Figure X. Agreement with WANDS human labels. The y-axis is narrowed to make differences between models easier to compare.*

![Severe error rate](/assets/img/2026-05-02-judgement-ai/model_severe_error_rate.png)

*Figure X. Severe error rate. Lower is better. A severe error means predicting `Exact` for a human `Irrelevant` item or `Irrelevant` for a human `Exact` item.*

The strongest judge was GPT-5.5:

| Model | Accuracy | Macro-F1 | Quadratic weighted kappa | MAE | Severe error rate |
|---|---:|---:|---:|---:|---:|
| GPT-5.5 | 0.738 | 0.738 | 0.705 | 0.284 | 0.022 |
| Qwen 4B fine-tuned, 620 steps | 0.642 | 0.623 | 0.576 | 0.382 | 0.024 |
| Qwen 4B instruct | 0.584 | 0.587 | 0.536 | 0.470 | 0.054 |

The fine-tuned Qwen model did improve over the instruct Qwen model, but the improvement was not dramatic. The best fine-tuned run, `qwen35-4b-bf16-wands620`, improved accuracy from `0.584` to `0.642`, macro-F1 from `0.587` to `0.623`, and quadratic weighted kappa from `0.536` to `0.576`.

The clearest improvement was in severe errors. I counted a severe error as a direct `Irrelevant` ↔ `Exact` mistake. The instruct Qwen model had a severe error rate of `5.4%`, while the best fine-tuned model reduced that to `2.4%`.

### Agreement vs severe mistakes

![Kappa vs severe error rate](/assets/img/2026-05-02-judgement-ai/kappa_vs_severe_error.png)

*Figure X. Judge quality as a tradeoff between agreement and severe mistakes. Better models are closer to the top-left.*

This plot made the tradeoff easier to see. GPT-5.5 had the best overall agreement. The fine-tuned Qwen model did not reach GPT-5.5 quality, but it moved in the right direction compared with the instruct model: higher agreement and fewer severe mistakes.

### What fine-tuning actually changed

![Recall by label](/assets/img/2026-05-02-judgement-ai/recall_by_label.png)

*Figure X. Recall by human label. The fine-tuned model became better at identifying Irrelevant and Partial examples, but weaker at confidently predicting Exact.*

The most interesting result was not simply “fine-tuning helped.” It was how fine-tuning helped.

The fine-tuned model became more conservative. It was much less likely to call an irrelevant product exact, but it also became much less likely to call an exact product exact.

For the best fine-tuned model:

- Human `Irrelevant` → predicted `Irrelevant`: `74.3%`
- Human `Partial` → predicted `Partial`: `76.5%`
- Human `Exact` → predicted `Exact`: only `35.7%`
- Human `Exact` → predicted `Partial`: `57.3%`

So the model learned caution. It often recognized that an exact product was at least relevant, but downgraded it to `Partial`.

### Bootstrap confidence intervals

I also used query-level bootstrap confidence intervals. This matters because the benchmark had 500 rows, but those rows came from only 50 queries. Rows from the same query are not fully independent.

The bootstrap results showed the same overall pattern:

| Model | Accuracy CI | Kappa CI | Severe error CI |
|---|---:|---:|---:|
| GPT-5.5 | 0.686–0.784 | 0.632–0.773 | 0.008–0.040 |
| Qwen 4B fine-tuned, 620 steps | 0.592–0.690 | 0.500–0.644 | 0.010–0.040 |
| Qwen 4B instruct | 0.524–0.648 | 0.451–0.626 | 0.032–0.078 |

This makes the result more cautious. The fine-tuned model improved in the raw metrics, especially on severe errors, but the confidence intervals are not wide enough to pretend this was a solved problem. The benchmark suggests improvement, not replacement-level reliability.

## Conclusion

This experiment did not prove that a small fine-tuned local model can replace human relevance judges.

It did show something more practical: a local model can be moved in the right direction with a small, focused fine-tuning dataset. The fine-tuned Qwen model was better than the instruct model, especially because it made fewer severe `Irrelevant` ↔ `Exact` mistakes.

But it was not close enough to call it a human replacement. Even GPT-5.5, the strongest judge in the benchmark, reached only `0.705` quadratic weighted kappa against the WANDS labels. That is useful, but not perfect. Relevance judging is still messy.

The fine-tuned model’s main weakness was that it became conservative. It often downgraded human `Exact` examples to `Partial`. That may be safer than over-predicting `Exact`, but it is still a problem if the goal is to replace human labels.

In hindsight, this was partly a skill issue on my side and partly the nature of the problem. Fine-tuning has a lot of degrees of freedom:

- which base model to use
- whether to start from a base or instruct model
- how to sample the dataset
- how to balance the labels
- how many epochs to train
- how to set the learning rate
- how to write the prompt
- whether to train only supervised fine-tuning or use preference/RL-style methods later

None of those choices has an obvious answer before running the experiment.

Either way, `judgement-ai` made it possible to run the same benchmark across local and hosted models, collect structured judgments, and compare them against human labels.

The final result is not “local LLMs replace human judges.”

The more honest result is:

> LLM judges are useful, but judging the judges is the real work.

For future work, I would not immediately repeat the same fine-tuning run. I would first improve the training data and evaluation loop. The most likely next steps would be better label balancing, using an instruction-tuned base model, adding disagreement analysis, and possibly trying preference or reinforcement-learning-style training. RL is still on the table, but only after the simpler supervised setup is better understood.
